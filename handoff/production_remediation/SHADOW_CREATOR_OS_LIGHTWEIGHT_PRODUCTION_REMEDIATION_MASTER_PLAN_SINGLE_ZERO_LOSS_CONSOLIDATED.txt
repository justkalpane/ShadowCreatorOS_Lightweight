SECTION 2 — SHADOW EMPIRE / FULL OS CONTEXT
================================================================================

2.1 ORIGINAL SHADOW EMPIRE AGENDA

Shadow Empire / Shadow Creator OS is a repo-driven intelligence operating system, not a generic prompt library.

Core intent:

- Build a founder-grade cognitive system for content creation and strategic execution.
- Convert user tasks into structured Shadow Mission Packets.
- Route missions through directors, agents, subagents, skills, subskills, tools, connectors, and provider handoff layers.
- Treat Context Engineering as the core output layer, not basic prompt writing.
- Preserve lineage, quality gates, auditability, and runtime boundaries.

2.2 ORIGINAL FULL OS ARCHITECTURE

The historical full OS included:

- 30 Directors as high-level strategic/intelligence owners.
- Large skill registry / skill.md contracts.
- Agents and subagents mapped to workflow lanes.
- n8n as orchestration/execution bus.
- Ollama/OpenWebUI/local reasoning sidecars in earlier Windows work.
- Data tables/state indexes for metadata.
- JSON/Markdown dossier artifacts for lineage and outputs.
- Provider handoff for voice/video/image/media/publishing/analytics.
- Strict PRD and governance laws: zero hallucination, zero deletion, append-only upgrades, coverage accounting, and evidence gates.

2.3 FULL OS DRIFT RISKS

The full/heavy runtime produced drift risk because:

- n8n active workflow IDs and webhooks can drift after imports/restarts.
- Provider credentials and API quotas introduce instability.
- OpenWebUI/auth/runtime dependencies increase operational burden.
- Old Windows runtime and DB profiles can be accidentally treated as active truth.
- Heavy provider/media execution can distract from the intelligence layer.
- Multiple runtimes increase debugging complexity.

2.4 WHY LIGHTWEIGHT PATH BECAME NECESSARY

The lightweight repo-first path exists to prove the intelligence layer can work immediately without heavy runtime dependencies:

User -> repo-aware coding agent/LLM -> repo-first routing -> director/agent/skill orchestration -> content/script/context engineering -> chat output.

n8n and providers are preserved as deferred execution layers only.
