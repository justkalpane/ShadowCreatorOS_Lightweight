# Gumloop Benchmark Output Standard

The Gumloop conversation is benchmark inspiration only. It is not the source of truth for Shadow OS routing.

## Patterns To Preserve

- Router loads first.
- Pipeline state machine is explicit.
- Trend/topic scoring is visible and qualified.
- Script engine produces 3 hook variants, emotion tags, readability/CTR-style estimates, and repair loops.
- Adversarial QA blocks unsupported execution claims.
- Voice, visual, editing, publishing, and provider boundaries are explicit.
- Missing integrations are disclosed instead of invented.

## Shadow OS Translation

- `KRISHNA` maps to local `Krishna` router evidence from `directors/supreme_vision/krishna.md`, `agents/krishna/krishna_agent.py`, and director registries.
- `ARUNA` maps to local `Aruna` flow evidence from `directors/kernel/aruna.md`, `agents/aruna/aruna_agent.py`, and runtime registries.
- Trend/topic intelligence maps through local Narada/Chanakya and topic/research skills.
- Script generation maps through local Vyasa/Valmiki/Saraswati plus script intelligence skills and subskills.
- QA maps through local governance, quality, policy, and adversarial review equivalents. Do not invent a missing Gumloop-only actor.

## Required Minimum Output

Every production content output must include:

- route activation proof
- director/skill consumption ledger
- topic gate
- 3 hook variants
- final script
- emotion/tone per beat
- timed beat map
- voice context
- image/video/edit/music context
- quality scorecard
- missed-rule ledger
- final verdict

## Assumption Control

If a score is estimated, label it as estimated.
If a source was not fetched, do not call the score source-verified.
If provider execution did not happen, do not claim generated audio, video, images, or published assets.
